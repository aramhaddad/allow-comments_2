Notes on: Introduction to Networks

A server is a computer that interacts with a request we make to the computer. For example, whenever we type in "http://www.google.com", we are sending a request to Google's servers to return a website.

Network: a group of entities that can communicate even though they are not all directly connected.

Channel: the way the entities talk to each other.

To have a network there has to be at least three entities, and there has to be a way where entities that are not directly connected can still communicate to each other.
Humans have been using networks over 3000 years old.

What do we need to make a network:
- We need a way to encode messages, we need to be able to convert the message and encode it in a signal.
- We need a way to do routing, where the message is going, and we need to know that it should be forwarded.

Flooding: sending the message everywhere.
- We need to decide who gets to use the network

Measuring Networks:
- Latency: The time it takes for a message to get from source to destination. Measured in units of time.
- Bandwidth: Amount of information that can be transmitted per unit time. It is measured by unit of information / unit of time.

What is a bit : the smallest unit of information. "0 or 1"
We can measure the bandwidth we get through our internet connection. We can go to CNET.com/internet-speed-test/ . And it will try sending some message to figure out what is my bandwidth.

To encode messages over the internet we use bits and over the wires we use electrons/photons.
To route message over the internet we use routers.
To decide who use the resources over the internet, there aren't rules except using the best effort service, there are policies that the routers use, and there is no guarantee as your packet might get dropped.
What we need to make a network work is a protocol.
A protocol is a set of rules that people agree to that tell you how two entities can talk to each other.

In the web the protocol gives rules on how the client "Web Browser" and the server communicate to each other.
The protocol that we use on the web is called hypertext transfer protocol HTTP.
Almost all the URLs start with HTTP indicating that the server you are contacting is using this protocol.

HTTP is a very simple protocol it has two messages:
- GET object and the server will run some code and get the file requested and then send that to you.
- Respond - contents of object.

HTTP is the main protocol used in the web, and it stands for Hyper Test Transfer Protocol.

The request from the browser begins with a GET request, example:
http://www.example.com/foo
GET /foo HTTP/1.1
HTTP is a very simple text protocol, and by sending GET /foo HTTP/1.1 the request is sent to the server.

The request line GET /foo HTTP/1.1 has three main parts:
- The Method "GET" : The Method is what type of request we are making to the server. The most common method we have is "GET" , and this is how we get a document from a server, other methods are used like POST and others. POST is often used for sending data to the server.
- The Path "/foo" : This is the actual document we are requesting from the server. This is the "relative path" to the server address www.example.com , and we can also specify the full path "http://example.com/foo"
- The Version "HTTP Version": it is always HTTP/ and then the version number. Most browsers and servers these days speak at 1.1

The Host name is not in the GET, the host is used to make the connection but we don't need it to get the document when we are connected to the server.
The HTTP request is followed by headers, and they have this format: Name: Value
Example: Get /foo?p=1 HTTP/1.1
Host: www.example.com This is useful because web servers may have multiple names
User-Agent: chrome This describes who is making the request, generally this is the browser
User-Agent is one of the most important headers because some people can show with fake users and they can pretend to be a browser, or some User-Agents can hurt some servers so knowing the user agent can help us adjust and fix the problems. Also we need to use user agents appropriately and be honest about our identity.

The HTTP Respond: the server respond with the document or information that you requested.

Notes on: Databases

Databases are how we actually store all the information we collect from users.
Database: A program that stores and retrieves "large amounts" of structured data.
Database can refer to a program that stores and retrieves data, to the machine running that program or to a group of machines working together to share/retrieve data.
Columns in the database are the properties that make up a particular type.

Every entry in the of a type in the table is called a row. A row is one instance of an element in a table.
Downside of querying data by hand:
- Error - prone
- Tedious
- Slow

Types of Databases:
• Relational, and it uses SQL : examples: Postgresql; MySQL; salite; Oracle… they all work with tables and they use SQL manipulate tabls.
• Google APP Engine's Data store.
• Dynamo by amazon.
• NoSQL: Example: Mongo; couch.

SQL: Structured Query Language. It is a language for expressing queries. So you would use SQL on a relational database to ask questions of the database to get data out of it. It was invented in the 1970s, long before the internet existed.
A basic SQL query would look like this:
Select * From links where id = s; select means "fetch" retrieve data. Star means all columns, links is the table we are fetching data from, and ID = 5 is a constraint.

Select * : this is what we are selecting, the * can be actually a list of columns.
The where id = s is the constrain , it is which rows from our table to return.
Module SQLite3 in python when imported it allows us to use SQLite in python.
The Where clause is not required when we want to fetch everything.

Joins: a SQL query that involved many tables. Join is a SQL statement that involves two tables.

Indexes:
Sequential scans, when we have sequent of something, and you search/find by going over the list and searching, but this is very slow for big lists.
Indexes is like an index in the box, it makes looking at things faster. (Key - Value) this makes queries run much faster.
Indexes for sorting:
Hash tables are not sorted, it is just a mapping of keys to values, and it is not sorted, it has constant time in looking ups.
There is a different type of mapping is called tree, and it is sorted, the lookups are slower. The lookups increases by increasing the size of the data table.

ACID:
Atomicity: All parts of a transaction "group of statements/ multiple commands" succeed or fail together.
Consistency: the database will always be consistent, the different lists should be updated and matches.
Isolation: no transaction can interfere with another.
Durability: once the transaction is committed, it won't be lost.
In the Google App Engine Data store, some vocabularies are different:
Tables ---> Entities in entities, columns are not fixed, entities in this are different than tables, all entities have an ID that can be assigned automatically. Entities has parent/ancestors notion.
